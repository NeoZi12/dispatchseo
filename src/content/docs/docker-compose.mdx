The way to self-host. One command starts everything - dashboard, MCP server,
database, schedules, and a headless Claude Code builder - as six small
containers using about 1 GB of RAM.

## Prerequisites

- [Docker](https://docs.docker.com/get-docker/) - Docker Desktop on Mac and
  Windows, or `docker` with the compose plugin on Linux.
- Nothing else. No cloud accounts of any kind.

## Install

```bash
git clone https://github.com/NeoZi12/dispatchseo &&
  cd dispatchseo &&
  sh start.sh
```

On Windows, run it in WSL or Git Bash. `start.sh` creates `.env`, generates
the one required secret, picks a free port, and starts the stack. The first
boot builds the app image (a few minutes) and applies the database schema
automatically. When it settles, open the URL it printed - usually
[http://localhost:3000](http://localhost:3000) - and the setup wizard walks
you through the rest.

Want your own database password? Set `POSTGRES_PASSWORD` in `.env` before
the very first start - it gets baked into the database volume.

## Laptop or always-on machine?

DispatchSEO works on schedules, and schedules only run while the machine is
on. A laptop is fine for trying it out: content builds catch up on their own
when the machine wakes (the builder checks what's due every 10 minutes), and
Search Console stats re-read the last few days on each pass. The one real
loss is rank history - positions are sampled live once a day, so a day the
machine slept through is a gap in the chart.

To run it seriously, use any machine that stays awake: a ~$5/month VPS with
1 GB of RAM, a Raspberry Pi, or a desktop that never sleeps. Nothing on the
internet needs to reach the machine, so a home computer behind a normal
router works without any port forwarding. On a VPS the install is the same
command over SSH - the
[self-hosting guide](https://github.com/NeoZi12/dispatchseo/blob/main/docs/SELF_HOSTING.md#install-on-a-vps)
has the step-by-step, including HTTPS with Caddy.

## What's running

| Container | Job |
| --- | --- |
| `app` | Dashboard, MCP server at `/api/mcp`, cron endpoints |
| `postgres` | Your data - persists in the `dispatch-pgdata` volume |
| `postgrest` | REST layer between app and database, internal network only |
| `migrate` | One-shot schema apply on every boot (idempotent, exits when done) |
| `cron` | Triggers rank checks, GSC snapshots, and weekly research on schedule |
| `builder` | Headless Claude Code - builds guides and tools, runs research (below) |

Check on it any time:

```bash
docker compose ps      # everything except the one-shot migrate should be running
docker compose logs app
docker compose logs builder
```

The `app` container reports `healthy` once it can reach the database through
the whole chain.

## The builder: automatic content, no public URL

The `builder` container is your own Claude Code running headlessly inside
the stack. Every 10 minutes it asks the backend what's due and runs it: the
daily guide build, weekly keyword research, approved tool builds, the weekly
AI-visibility scan - and on auto-mode projects it merges green guide PRs. It
only makes outbound connections (to GitHub and Anthropic), which is why a
localhost install is fully automatic with no tunnel or domain.

Turning it on is one token:

1. On your own computer, run `claude setup-token` and copy the
   `sk-ant-oat...` token. Builds run on your existing Claude subscription.
2. Add it to `.env`: `CLAUDE_CODE_OAUTH_TOKEN=sk-ant-oat...`
3. `docker compose up -d builder`

GitHub access reuses the token from the wizard's One-tap merge step. Until
the Claude token is set, the container idles and
`docker compose logs builder` says what it's waiting for.

## When do you need a public URL?

Only if you skip the builder and want the GitHub-hosted workflow schedules
(the files the install puts in your site's repo) to do the building instead -
those run on GitHub's servers and must call back to your instance. In that
case put the instance behind a reverse proxy with HTTPS or a tunnel like
Cloudflare Tunnel, and set it in `.env`:

```bash
APP_URL=https://dispatch.your-domain.com
```

Restart with `sh start.sh` after changing `.env`. PR checks (like tool
validation) run on GitHub either way and work regardless.

## Configuration

Only `CRON_SECRET` is required, and `start.sh` generates it. Everything
else - dashboard password, MCP token, Search Console, DataForSEO - is
handled in the dashboard's setup wizard, not in env vars. The optional
overrides (and what they do) are annotated in
[`.env.docker.example`](https://github.com/NeoZi12/dispatchseo/blob/main/.env.docker.example).

Useful ones:

```bash
DISPATCH_PORT=3000   # host port the dashboard is published on
APP_URL=             # public URL of this instance (see above)
```

## Upgrading

```bash
git pull && sh start.sh
```

The script rebuilds what changed; schema changes apply automatically on
boot - the migration file is idempotent, so re-running it is always safe.
Your data lives in the `dispatch-pgdata` volume and survives rebuilds.

Back up before big upgrades:

```bash
docker compose exec postgres pg_dump -U dispatch dispatchseo > backup.sql
```

## Uninstall

```bash
docker compose down        # stop, keep data
docker compose down -v     # stop and DELETE ALL DATA
```
