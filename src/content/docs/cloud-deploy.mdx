The zero-server path: Vercel (Hobby) runs the app, Supabase (free) holds the
data, GitHub Actions runs the schedules. Hosting costs $0 - you bring your
own free accounts.

## Deploy the backend

Click a deploy button on the
[repo README](https://github.com/NeoZi12/dispatchseo#readme) - neither asks
you for anything:

- **Deploy - new database** clones the repo to your GitHub and provisions a
  free Supabase database through the Vercel Marketplace (its env vars get
  injected automatically).
- **Deploy - your own database** deploys the same app without creating a
  database. Use it if your Supabase account is at its free-project limit
  (the free tier allows 2 active projects) or you want to reuse an existing
  project.

Prefer doing it by hand? Fork the repo, then import the fork into Vercel
(Add New → Project). That's equivalent to the second button.

## Let the wizard finish it

Open your new deployment. The setup wizard shows exactly what's left, in
order:

1. **Connect your database** (skipped on the marketplace path): copy your
   Supabase project's URL and secret key into the Vercel env vars
   `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`, then redeploy. The
   wizard shows these steps with links.
2. **Run the migrations**: paste
   [`supabase/migrations/setup.sql`](https://github.com/NeoZi12/dispatchseo/blob/main/supabase/migrations/setup.sql)
   - every migration in one file, safe to re-run - into the Supabase SQL
   Editor and press Run. The wizard verifies the full set actually ran
   before letting you continue.
3. **Claim the instance**: choose your dashboard password. DispatchSEO
   generates its own MCP token and cron key and shows you both.

## Enable the schedules

Vercel Hobby caps crons at once per day, so the schedules are split:

- **Vercel cron** runs the daily rank check - already configured in
  `vercel.json`, nothing to do.
- **GitHub Actions** runs the higher-frequency jobs. On your fork: enable
  Actions (forks start disabled), then set
  - repository **secret** `CRON_SECRET` - your cron key from the wizard
    (shown at `/setup/keys` on your deployment),
  - repository **variable** `BACKEND_URL` - your deployment URL.

Only `hourly-gsc.yml` matters for a fresh install. The `seo-*.yml` workflows
in the repo are its own content pipeline - your site gets its own copies
when you [connect it](/docs/connect-your-site), so leave those disabled on
your fork.

## Environment variables

The wizard handles the required ones. Everything optional is annotated in
[`.env.local.example`](https://github.com/NeoZi12/dispatchseo/blob/main/.env.local.example)
and goes in Vercel under Project → Settings → Environment Variables
(redeploy after saving).

| Variable | What it is |
| --- | --- |
| `GSC_SERVICE_ACCOUNT_JSON` | [Search Console setup](/docs/search-console) - one line, the whole JSON |
| `GSC_SITE_URL` | Your property, e.g. `sc-domain:example.com` |
| `RESEND_API_KEY` + `ALERT_EMAIL` | Failure emails - strongly recommended in auto mode |
| `DATAFORSEO_LOGIN` / `DATAFORSEO_PASSWORD` | Optional paid data - volumes, difficulty, domain rating |
